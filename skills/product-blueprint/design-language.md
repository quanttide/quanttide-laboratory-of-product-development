# 设计语言 · 布局原则

> 源自"墨弈"产品蓝图循环中的设计迭代沉淀。

---

## 核心原则

### 1. 共享控件保持中立

属于双方的操作，不挂在任何一方面板下。

- 对局控制（新对局/悔棋/虚着）、棋盘尺寸、快捷键等与特定玩家无关的控件，置于中立位置
- 不要隐含"这个控件属于黑方或白方"的错误信号

### 2. 棋盘是视觉焦点，控制不是

棋盘独占中心区域，操作控件退居次要层级。

- 控件不放在棋盘下方（破坏视觉重心）
- 工具条模式（棋盘上方）让用户先扫操作选项，再沉入棋盘
- 棋盘下方只保留轻量状态提示

### 3. 左右对称是默认基线

不破坏对称的设计必须有明确理由。

- 两侧面板承载对等的玩家信息，视觉重量均衡
- 如果一侧多放了内容（如规则说明），另一侧也要有对等物补偿
- 对称被破坏时，用户会感知到"歪了"

### 4. 物理隐喻引导布局

数字界面布局可以借鉴物理世界的参考。

- 围棋棋盘放桌上，双方分坐两侧，共用棋盘和棋罐
- 工具条类比"桌边的棋罐和计时器"——独立于玩家区域，双方伸手可及
- 用这个隐喻判断：这个东西在物理棋盘上会出现在哪里？

---

## 布局层级（自上而下）

```
1. 标题区     → 品牌识别、对局编号、棋盘尺寸标识
2. 工具条     → 共享控件（中立、双方共用）
3. 棋盘区     → 三栏：黑方 | 棋盘+状态 | 白方
4. 棋谱区     → 对局记录
5. 底部       → 装饰性信息
```

## 适用检查清单

- [ ] 共享控件是否挂在某个玩家面板下？→ 移到中立位置
- [ ] 棋盘下方是否有操作控件？→ 移到上方或侧边
- [ ] 左右面板视觉重量是否明显不对称？→ 补充或裁剪
- [ ] 布局是否符合物理世界的直觉？→ 如果不，原因是否充分
