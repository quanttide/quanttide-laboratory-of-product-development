# 技术视角 · 技术架构描述

> 核心问题：做得对不对？

---

## 1. 架构概览

一张文字架构图或一句话概括系统的架构风格。

- 架构风格：单页面应用（SPA）、微前端、服务端渲染、静态站点等
- 分层描述：展示层、逻辑层、数据层如何划分
- 外部依赖：使用了哪些外部服务（CDN、字体、API 等）

**输出格式**：一句话架构概括 + 分层列表 + 外部依赖清单。

## 2. 组件关系与技术栈

列出系统的各个模块/组件及其技术实现。

每个模块格式：

- **模块名**：职责描述
- **技术选型**：使用的库/框架/工具
- **依赖关系**：依赖了哪些其他模块
- **关键接口/约定**：模块间通信方式（事件、回调、全局状态等）

## 3. 数据流与状态管理

描述数据在系统中如何流转。

- 状态存储位置：全局状态 vs 局部状态 vs DOM 状态
- 状态变更路径：用户操作 → 事件处理 → 状态更新 → 视图渲染
- 关键数据结构：核心数据模型的形状与字段含义
- 持久化策略：数据是否持久化、存储方式（LocalStorage、IndexedDB 等）

**输出格式**：数据流图（文字描述） + 核心数据结构定义 + 持久化方案。

## 4. 架构决策记录 (ADR)

列出重要的技术决策和取舍。

每条决策格式：

- **标题**：决策名称
- **上下文**：面临的问题或约束
- **方案**：选择了什么技术/模式
- **理由**：为什么选这个
- **后果**：接受的风险/限制/后续成本
- **替代方案**：考虑过但放弃的方案及原因

## 5. 性能与安全

- 性能策略：加载优化、渲染优化、缓存策略
- 安全边界：输入验证、内容安全策略、依赖安全
- 监控与日志：错误追踪、用户行为采集

## 完成检查

确认以下问题都已回答：架构风格与技术栈是什么？组件如何划分与通信？数据如何流转与存储？关键架构决策是否有记录？如有缺失，在末尾用一句话说明。
